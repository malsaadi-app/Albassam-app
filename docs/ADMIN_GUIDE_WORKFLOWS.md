## 🎯 نظرة عامة

### ما هو نظام سير العمل؟

نظام يتيح لك:
- ✅ تصميم مسارات الموافقات
- ✅ تحديد المسؤولين عن كل مرحلة
- ✅ إدارة القواعد والشروط
- ✅ تتبع الأداء والإحصائيات

### المفاهيم الأساسية

#### **1. Workflow (سير العمل)**
- مجموعة خطوات متسلسلة
- مثال: "سير عمل طلب إجازة"

#### **2. Step (الخطوة)**
- مرحلة موافقة واحدة
- مثال: "موافقة المدير المباشر"

#### **3. Version (الإصدار)**
- نسخة من سير العمل
- حالات: مسودة / منشور

#### **4. Runtime (التشغيل)**
- تنفيذ فعلي لسير عمل
- يتم عند تقديم طلب

**املأ النموذج:**

| الحقل | الوصف | مثال |

| **الاسم** | اسم سير العمل بالعربي | طلب إجازة اعتيادية |

### الخطوة 3: إضافة خطوات الموافقة

#### **خطوة موافقة - مثال 1: المدير المباشر**

```
┌────────────────────────────────────┐
│ اسم الخطوة:  موافقة المدير المباشر│
│ الترتيب:     1                     │
│ المستوى:     STAGE (مستوى المرحلة) │
│ الموافق:     DIRECT_MANAGER        │
│ شرط:        ---                    │
│ التصعيد:    3 أيام                 │
└────────────────────────────────────┘
```

**الحقول:**
- **اسم الخطوة:** وصف المرحلة
- **الترتيب:** 1, 2, 3... (الأولوية)
- **المستوى:**
  - STAGE: مستوى المرحلة
  - DEPARTMENT: مستوى القسم
  - COMPANY: مستوى الشركة
- **نوع الموافق:**
  - SPECIFIC_USER: موظف محدد
  - ROLE: دور معين (مدير HR)
  - DIRECT_MANAGER: المدير المباشر
  - SYSTEM_ROLE: دور نظام (ADMIN)
- **الشرط (اختياري):** `amount > 5000` (للمبالغ الكبيرة)
- **التصعيد التلقائي:** عدد الأيام قبل التصعيد

---

#### **خطوة موافقة - مثال 2: مدير الموارد البشرية**

```
┌────────────────────────────────────┐
│ اسم الخطوة:  موافقة مدير HR        │
│ الترتيب:     2                     │
│ المستوى:     DEPARTMENT            │
│ الموافق:     ROLE → hr_manager     │
│ شرط:        days > 7               │
│ التصعيد:    2 أيام                 │
└────────────────────────────────────┘
```

**الشرط:** `days > 7`  
**المعنى:** هذه الخطوة تُنفذ فقط إذا كانت الإجازة أكثر من 7 أيام

---

#### **خطوة موافقة - مثال 3: المدير العام**

```
┌────────────────────────────────────┐
│ اسم الخطوة:  موافقة المدير العام   │
│ الترتيب:     3                     │
│ المستوى:     COMPANY               │
│ الموافق:     SPECIFIC_USER → CEO   │
│ شرط:        days > 15              │
│ التصعيد:    1 يوم                  │
└────────────────────────────────────┘
```

**الشرط:** `days > 15`  
**المعنى:** إجازات أكثر من 15 يوم تحتاج موافقة المدير العام

### الخطوة 4: إعدادات الإشعارات

لكل خطوة، حدد:

| الإشعار | الوصف | التوصية |
|---------|------|---------|
| **عند الدخول** | إرسال إشعار عند وصول الطلب | ✅ مفعل |
| **عند الموافقة** | إرسال إشعار بعد الموافقة | ✅ مفعل |
| **عند الرفض** | إرسال إشعار بعد الرفض | ✅ مفعل |

---

### الخطوة 5: الحفظ والنشر

#### **الحفظ كمسودة:**
```
💾 حفظ → DRAFT
```
- يمكنك التعديل لاحقاً
- لا يظهر في الإنتاج

#### **النشر:**
```
🚀 نشر → PUBLISHED
```
- يصبح نشط فوراً
- لا يمكن تعديله (إنشاء إصدار جديد)
- يستخدم في الطلبات الجديدة

## ✏️ تحرير سير عمل

### تعديل مسودة:

1. اذهب إلى **سير العمل**
2. ابحث عن المسودة
3. اضغط **"تحرير"**
4. عدّل ما تريد
5. احفظ أو انشر

---

### تعديل سير عمل منشور:

⚠️ **لا يمكن تعديل سير عمل منشور مباشرة!**

**السبب:** للحفاظ على الطلبات الحالية

**الحل: إنشاء إصدار جديد**

#### الخطوات:

1. افتح سير العمل المنشور
2. اضغط **"إصدار جديد"**
3. سيُنشأ نسخة جديدة (مسودة)
4. عدّل النسخة الجديدة
5. احفظها أو انشرها

#### ماذا يحدث بعد النشر؟

```
الإصدار القديم (v1):
└─ الطلبات الحالية تستمر عليه

الإصدار الجديد (v2):
└─ الطلبات الجديدة تستخدمه
```

## 📊 إدارة الإصدارات

### عرض الإصدارات

```
سير العمل → [اسم السير] → الإصدارات
```

**ستجد:**

| الإصدار | الحالة | الطلبات | التاريخ |
|---------|--------|---------|---------|
| v3 | منشور | 45 | 2026-03-10 |
| v2 | غير نشط | 122 | 2026-02-15 |
| v1 | غير نشط | 78 | 2026-01-10 |

---

### إلغاء تفعيل إصدار

⚠️ **تحذير:** سيؤثر على الطلبات الجديدة فقط

**الخطوات:**
1. اذهب إلى سير العمل
2. اضغط على **"..."** → **"إلغاء التفعيل"**
3. أكد القرار

**النتيجة:**
- الطلبات الجديدة: لن تستخدم هذا السير
- الطلبات الحالية: تستمر عادي

---

### حذف مسودة

1. اذهب إلى المسودة
2. اضغط **"..."** → **"حذف"**
3. أكد الحذف

⚠️ **لا يمكن حذف سير عمل منشور!**

## 🔧 استكشاف الأخطاء

### المشكلة 1: سير العمل لا يبدأ تلقائياً

**الأعراض:**
- طلب مُقدم لكن لا يظهر في الموافقات

**الحلول:**

1. **تحقق من النشر:**
   ```
   سير العمل → [اسم السير] → الحالة
   يجب أن تكون: PUBLISHED ✅
   ```

2. **تحقق من الفروع/المراحل:**
   - هل الطلب من فرع/مرحلة مشمولة؟

3. **تحقق من النوع:**
   - هل نوع الطلب يطابق سير العمل؟

4. **راجع السجلات:**
   ```
   الإعدادات → السجلات → سير العمل
   ```

---

### المشكلة 2: الطلب عالق في مرحلة

**الأعراض:**
- الطلب منتظر منذ أيام
- لا أحد يستلم إشعار

**الحلول:**

1. **تحقق من الموافق:**
   ```
   افتح الطلب → تفاصيل الموافقة
   من هو الموافق المحدد؟
   ```

2. **هل الموافق لديه صلاحية؟**
   ```
   الإعدادات → المستخدمين → [الموافق] → الأدوار
   ```

3. **التصعيد اليدوي:**
   ```
   افتح الطلب → خيارات → تصعيد للمرحلة التالية
   ```

4. **إعادة تعيين الموافق:**
   ```
   افتح الطلب → خيارات → تغيير الموافق
   ```

---

### المشكلة 3: الإشعارات لا تصل

**الأعراض:**
- الموافق لا يستلم إشعارات

**الحلول:**

1. **تحقق من إعدادات الإشعارات:**
   ```
   سير العمل → [الخطوة] → الإشعارات
   يجب أن تكون مفعلة ✅
   ```

2. **تحقق من إعدادات المستخدم:**
   ```
   المستخدم → الإعدادات → الإشعارات
   ```

3. **اختبر الإشعار:**
   ```
   الإعدادات → الإشعارات → إرسال تجريبي
   ```

---

### المشكلة 4: خطأ في الشرط

**الأعراض:**
- الخطوة لا تُنفذ رغم تحقق الشرط

**الحلول:**

1. **راجع صياغة الشرط:**
   ```
   ❌ خطأ: days > 7 days
   ✅ صحيح: days > 7

   ❌ خطأ: amount more than 5000
   ✅ صحيح: amount > 5000
   ```

2. **الشروط المدعومة:**
   ```
   - amount > 5000
   - days >= 7
   - days < 15
   - priority == "URGENT"
   - branch == "riyadh"
   ```

3. **اختبر الشرط:**
   ```
   سير العمل → [الخطوة] → اختبار الشرط
   ```

## 💡 أفضل الممارسات

### ✅ افعل:

1. **ابدأ بسيط:**
   - سير عمل بسيط (2-3 خطوات)
   - اختبره جيداً
   - ثم أضف التعقيد

2. **اختبر قبل النشر:**
   - أنشئ طلبات تجريبية
   - تأكد من المسار الكامل
   - راجع الإشعارات

3. **وثّق القرارات:**
   - لماذا هذه الخطوة؟
   - لماذا هذا الشرط؟
   - اكتب في الوصف

4. **راقب الأداء:**
   - تحقق من التحليلات أسبوعياً
   - حسّن الاختناقات
   - استمع لملاحظات الموظفين

5. **حدّث بانتظام:**
   - راجع سير العمل كل 3 شهور
   - هل لا يزال مناسباً؟
   - احذف ما لا يُستخدم

---

### ❌ لا تفعل:

1. **لا تعدل سير عمل منشور مباشرة:**
   - ستؤثر على الطلبات الحالية
   - أنشئ إصدار جديد دائماً

2. **لا تضع خطوات كثيرة:**
   - سير العمل الطويل = تأخير
   - اهدف لـ 2-4 خطوات فقط

3. **لا تنسَ التصعيد:**
   - الطلب قد يتأخر
   - ضع حد زمني معقول (2-5 أيام)

4. **لا تعقّد الشروط:**
   - شروط معقدة = أخطاء
   - اجعلها بسيطة وواضحة

5. **لا تهمل التدريب:**
   - الموافقون يحتاجون تدريب
   - وفّر دليل مستخدم واضح

## 📚 أمثلة عملية

### مثال 1: سير عمل بسيط (إجازة اعتيادية)

```yaml
الاسم: طلب إجازة اعتيادية
النوع: HR_REQUEST → LEAVE_REQUEST
الشرط: days <= 7

الخطوات:
  1. موافقة المدير المباشر
     - المستوى: STAGE
     - الموافق: DIRECT_MANAGER
     - التصعيد: 3 أيام

النتيجة:
  - طلبات بسيطة (≤7 أيام)
  - خطوة واحدة فقط
  - سريع وفعال
```

---

### مثال 2: سير عمل متوسط (إجازة طويلة)

```yaml
الاسم: طلب إجازة طويلة
النوع: HR_REQUEST → LEAVE_REQUEST
الشرط: days > 7 AND days <= 15

الخطوات:
  1. موافقة المدير المباشر
     - المستوى: STAGE
     - الموافق: DIRECT_MANAGER
     - التصعيد: 2 أيام

  2. موافقة مدير الموارد البشرية
     - المستوى: DEPARTMENT
     - الموافق: ROLE → hr_manager
     - التصعيد: 2 أيام

النتيجة:
  - إجازات 8-15 يوم
  - خطوتان
  - توازن بين السرعة والرقابة
```

---

### مثال 3: سير عمل معقد (مشتريات كبيرة)

```yaml
الاسم: طلب شراء كبير
النوع: PROCUREMENT_REQUEST
الشرط: amount > 50000

الخطوات:
  1. موافقة مدير المشتريات
     - المستوى: DEPARTMENT
     - الموافق: ROLE → procurement_manager
     - التصعيد: 1 يوم

  2. موافقة المدير المالي
     - المستوى: COMPANY
     - الموافق: ROLE → cfo
     - التصعيد: 1 يوم

  3. موافقة المدير العام
     - المستوى: COMPANY
     - الموافق: SPECIFIC_USER → CEO
     - التصعيد: 1 يوم

النتيجة:
  - مشتريات > 50,000 ريال
  - 3 خطوات موافقة
  - رقابة صارمة
```
